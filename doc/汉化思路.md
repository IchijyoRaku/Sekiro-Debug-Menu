# 调试菜单文本汉化思路

## 1. 结论

是的，当前菜单显示的文本可以判断为**游戏程序内部原本就存在的调试菜单文本**，而不是本项目在 [`DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 中自行定义出来的字符串。

判断依据主要有两点：

1. [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 的参数里直接接收了游戏传进来的 `wchar_t* pwString`。
2. 该函数并没有构造新文本，只是把 `pwString` 复制进 [`SDebugPrintStruct`](dinput8/SekiroDebugMenu/Graphics/Graphics.h:11) 缓冲区，随后由 [`CGraphics::DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 原样绘制。

也就是说，这个项目做的事情更像是：

- 截获游戏原本想画出来的调试菜单文本
- 记录文本内容和坐标
- 再用 ImGui 覆盖层重绘一次

因此，**文本源头在游戏内存/游戏代码里，本项目只是“搬运并显示”**。

---

## 2. 当前文本流转路径

从源码看，文本显示链路如下：

1. 游戏内部某处调试菜单逻辑生成文本。
2. 游戏调用被 Hook 的菜单绘制位置。
3. [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 拿到：
   - 坐标 [`SMenuDrawLocation`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.h:4)
   - 文本指针 `wchar_t* pwString`
4. Hook 把文本复制到 [`sDebugMenuStruct`](dinput8/SekiroDebugMenu/Graphics/Graphics.h:41) 中。
5. [`CGraphics::DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 再读取这些文本并渲染。

因此，做汉化最合适的切入点，通常不是去改 ImGui 绘制逻辑本身，而是**在文本进入绘制缓冲区之前或进入之后做替换**。

---

## 3. 汉化可以采用的几种思路

下面按照实现难度和维护成本，从低到高分析。

### 方案一：在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 中做“字符串映射替换”

这是最直接、最适合当前项目结构的方式。

#### 3.1 实现思路

在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 里，拿到 `pwString` 后，不立刻原样写入缓冲区，而是：

1. 先把 `pwString` 转成 [`std::wstring`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:197)
2. 在一个“英文 -> 中文”的映射表里查找
3. 如果命中，就写入中文
4. 如果未命中，就保留原文

#### 3.2 适合场景

适合：

- 调试菜单文本数量有限
- 字符串相对固定
- 希望快速落地
- 不想深入改游戏内存或游戏资源

#### 3.3 优点

- 改动集中，风险低
- 不需要重新逆向游戏字符串资源位置
- 与现有项目耦合度高，容易维护
- 可以逐步补充翻译表

#### 3.4 缺点

- 只能翻译已经被 Hook 捕获到的文本
- 若文本里有动态拼接内容，需要额外规则处理
- 本质上属于“显示层替换”，不是底层资源汉化

#### 3.5 建议实现结构

可以新建一个模块，例如：

- [`dinput8/SekiroDebugMenu/Localization/Localization.h`](dinput8/SekiroDebugMenu/SekiroDebugMenu.h)
- [`dinput8/SekiroDebugMenu/Localization/Localization.cpp`](dinput8/SekiroDebugMenu/SekiroDebugMenu.cpp)

提供类似接口：

- `const wchar_t* TranslateDebugText(const wchar_t* original)`
- 或 `std::wstring TranslateDebugText(const std::wstring& original)`

然后在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 中调用。

#### 3.6 推荐数据形式

可使用：

- `std::unordered_map<std::wstring, std::wstring>` 做精确匹配
- 对动态文本再补充前缀匹配、关键词匹配或规则替换

示例思路：

- `"Graphics Debug" -> "图形调试"`
- `"Free Camera" -> "自由视角"`
- `"ON" -> "开启"`
- `"OFF" -> "关闭"`

这是我认为**最优先推荐**的方案。

---

### 方案二：在 [`CGraphics::DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 绘制前做替换

这个方案和方案一很接近，但替换时机后移到了 ImGui 绘制阶段。

#### 3.1 实现思路

在 [`DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 中遍历 [`SDebugPrintStruct`](dinput8/SekiroDebugMenu/Graphics/Graphics.h:11) 时，把 `pStruct->wText` 先经过翻译函数，再送入 `AddText`。

#### 3.2 优点

- 不改 Hook 逻辑，改动位置直观
- 便于在同一处统一处理编码与字体显示问题

#### 3.3 缺点

- 文本缓冲区里仍然保存的是原文
- 若后续还有别处复用缓冲区数据，拿到的还是英文
- 从职责上讲，绘制层开始兼顾翻译，语义略杂

#### 3.4 适合场景

如果你想尽量少碰 Hook 代码，或者想先快速验证字体与中文显示链路是否通畅，可以先在这里做。

---

### 方案三：直接修改游戏内存中的字符串

这个方案更底层，也更像真正意义上的“运行时汉化”。

#### 3.1 实现思路

通过以下方式定位游戏内的原始文本来源：

1. 利用当前已知的 `pwString` 指针，观察它指向的内存区域
2. 判断这些字符串是：
   - 静态常量字符串
   - 动态拼接缓冲区
   - 某种资源表/文本表
3. 找到其上游生成位置后，直接：
   - 改写字符串常量
   - 改写字符串表
   - 或 Hook 生成函数，返回汉化结果

#### 3.2 优点

- 更彻底，理论上可以让游戏原逻辑层就拿到中文
- 如果菜单在其他地方也会用到同一文本，可一并生效

#### 3.3 缺点

- 逆向成本明显更高
- 字符串长度变化可能引发风险
- 若原本是窄字符、内部编码特殊或动态拼装，处理会更复杂
- 维护成本高，对游戏版本更敏感

#### 3.4 适合场景

适合后续深入研究，但**不建议作为第一步**。

---

### 方案四：基于地址或调用上下文做精细化翻译

有些菜单项可能文字相同，但在不同页面里含义不同；或者菜单文字由多个短词组合而成。这时可以考虑“上下文化翻译”。

#### 4.1 实现思路

在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 中，除了文本外，还结合：

- 文本坐标 `fX` / `fY`
- 调用来源地址
- 当前菜单层级
- 前后连续出现的文本

来判断该文本属于哪个菜单区域，再做不同翻译。

#### 4.2 优点

- 适合复杂菜单结构
- 能处理同词多义

#### 4.3 缺点

- 需要维护更多上下文逻辑
- 复杂度高于简单映射表

这个方案通常作为方案一的增强版，而不是替代版。

---

## 4. 最推荐的落地路线

结合当前项目实现，推荐采用下面这条路线：

### 第一步：先保证中文字体可显示

当前 [`CGraphics::D3D11PresentHook()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:99) 中会加载 [`debugfont.ttf`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:153)。

需要优先确认：

1. 游戏目录下是否存在该字体文件
2. 该字体是否包含常用中文字符
3. 当前 ImGui 字形范围是否足够覆盖简体中文

现有 `RANGES` 已包含 CJK Unified Ideographs 区间，因此从代码上看是支持中文字符范围的；关键在于字体文件本身是否真的有中文字形。

如果 [`debugfont.ttf`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:153) 不包含中文，就算替换成功也会显示为方块或空白。

#### 推荐字体思路

可换成支持中文的字体，例如：

- 思源黑体 / Noto Sans CJK
- 微软雅黑
- 更轻量的中文像素或 UI 字体

若字体体积过大，也可以后续考虑字集裁剪。

---

### 第二步：先做“显示层翻译表”

优先在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 或 [`DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 增加翻译映射，这是投入产出比最高的方案。

建议顺序：

1. 先记录所有实际出现的英文菜单项
2. 整理成对照表
3. 实现精确匹配翻译
4. 再补充动态规则

这个阶段就能很快得到一个“可用的中文调试菜单”。

---

### 第三步：增加日志，导出原始菜单文本

为了高效整理翻译表，建议增加日志功能。

#### 建议做法

在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 中：

- 把每次出现的新字符串输出到文件
- 或输出到调试控制台
- 只记录未见过的条目，避免刷屏

这样你在实际打开各级调试菜单时，就能自动收集词条。

#### 日志用途

- 建立英中对照表
- 统计是否存在重复文本
- 发现动态文本模板
- 检查是否有乱码/编码问题

这是汉化前非常值得做的一步。

---

### 第四步：再决定是否做底层汉化

如果只是为了“自己使用调试菜单”，做到第二步通常已经足够。

如果你的目标是：

- 做成完整汉化补丁
- 对菜单系统进行更深层改造
- 尽量让原始逻辑链路也用中文

再去研究“字符串来源地址”和“原始资源表”会更合适。

---

## 5. 实际开发时的几个关键注意点

### 5.1 中文文本长度会更长

当前 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:14) 使用 `memcpy(pStruct->wText, pwString, 512);`，这是一个比较粗糙的复制方式。

而 [`SDebugPrintStruct::wText`](dinput8/SekiroDebugMenu/Graphics/Graphics.h:15) 实际大小是 1200 个 `wchar_t`。

这里至少要注意：

- 原代码按字节复制固定长度，不一定保证字符串结尾安全
- 如果你替换成更长中文文本，最好改成更安全的宽字符复制方式
- 应保证以 `L'\0'` 结尾

所以汉化时，建议顺带把这一段改成更稳妥的字符串写入逻辑。

### 5.2 编码统一使用 UTF-16 / UTF-8

项目内部原始文本是 `wchar_t*`，在 Windows 下通常是 UTF-16 宽字符。

而 ImGui 绘制使用的是 UTF-8 字符串，因此 [`DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:200) 中已经做了一次宽字符到 UTF-8 的转换。

这意味着：

- 翻译表内部如果使用 [`std::wstring`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:197)，会比较自然
- 最终仍由现有转换逻辑输出给 ImGui

因此建议整个汉化层统一使用宽字符存储。

### 5.3 字体渲染范围与性能

完整中文字体较大，加载时间和纹理体积都会增加。

如果后期要优化，可以：

- 只保留调试菜单实际用到的字集
- 用自定义 glyph ranges 精简字符范围
- 用较轻量的中文字体替换

### 5.4 同一英文词可能在不同场景语义不同

比如 `Enable`、`Disable`、`Reset`、`Default` 这类通用词，如果直接逐词替换，有时会损失上下文信息。

更稳妥的方式是优先翻译完整菜单项，而不是先做单词级替换。

---

## 6. 推荐的实施方案总结

如果目标是“把当前调试菜单尽快做成可用中文版本”，推荐优先级如下：

### 优先级 A：最值得先做

1. 确认 [`debugfont.ttf`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:153) 支持中文
2. 在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 或 [`DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 接入翻译表
3. 增加日志，自动收集实际菜单词条

### 优先级 B：做完善

4. 增加动态文本规则翻译
5. 按菜单上下文优化翻译准确性
6. 调整排版，避免中文过长造成遮挡

### 优先级 C：做深入逆向

7. 逆向原始字符串来源
8. 直接修改游戏内部字符串表或生成逻辑

---

## 7. 最终建议

从当前项目结构看，**最现实、成本最低、成功率最高的汉化方式，是在 [`DrawMenuHook()`](dinput8/SekiroDebugMenu/Hooks/Patches/Patches.cpp:3) 或 [`CGraphics::DrawMenu()`](dinput8/SekiroDebugMenu/Graphics/Graphics.cpp:184) 中加入一个“英文到中文的翻译映射层”**。

这是因为：

- 文本已经被稳定截获
- 渲染链路已经完整打通
- 项目已支持宽字符到 UTF-8 的转换
- 只要字体支持中文，就可以较快做出可用结果

所以推荐的核心思路可以概括为：

**先确认字体，再收集词条，再做映射翻译，最后视需要再深入到底层资源汉化。**
